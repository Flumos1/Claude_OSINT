# OSINT Orchestrator

Дирижёр расследования. Не собирает данные сам — структурирует задачу и маршрутизирует.

## Шаг 1. Рамка и основание (обязательно)

Перед сбором уточни и зафиксируй:
- **Объект:** что/кто проверяется (название, домен, ФИО, email, IP...).
- **Цель (IR):** какие конкретные вопросы закрываем. Сформулируй список.
- **Основание:** зачем, есть ли законное основание (особенно для физлиц).

Сверься с рамками в [CLAUDE.md](../../../CLAUDE.md) и [knowledge/ethics-legal.md](../../../knowledge/ethics-legal.md).
Если объект — частное лицо без законного основания → остановись и проговори с пользователем.

## Шаг 2. Завести кейс

Создай `cases/<slug>/` копией `cases/_TEMPLATE/`. Заполни `00-brief.md` (объект, IR, основание, дата).
Веди `log.md` по ходу.

## Шаг 3. Определить тип объекта → скил

| Объект | Скил |
|--------|------|
| Компания, юрлицо, контрагент, бенефициары | **company-dd** |
| Человек, ФИО, email, username, телефон (с основанием) | **person-osint** |
| Домен, сайт, IP, инфраструктура, поддомены | **domain-infra** |
| Утечка, фишинг, IOC, репутация IP/файла, мониторинг бренда | **threat-intel** |
| Несколько типов | прогони последовательно, связывай результаты |

## Шаг 4. Pivoting между объектами

Объекты связаны: компания → её домен (domain-infra) → утечки (threat-intel) →
бенефициар (person-osint). Каждый найденный селектор может породить новую ветку.
Держи карту связей в `cases/<slug>/entities.md`. См. [knowledge/methodology.md](../../../knowledge/methodology.md) (Pivoting).

## Шаг 5. Сборка и отчёт

Когда IR закрыты или источники исчерпаны → скил **osint-report** для финального отчёта.

## Принципы (напоминание)

- Каждый факт — с источником и датой (provenance).
- Разделяй ФАКТ / ВЫВОД / ГИПОТЕЗА, маркируй достоверность ([A1]..[F6]).
- Минимизация: только то, что нужно под IR.
- Архивируй уходящие данные (Wayback/локально).
